## Findings (deduplicated, sorted by severity)

### Critical

1. **Command injection risk in workflow input interpolation**
- File: `.github/workflows/backtest.yml` (lines 79-83, 111, 154-157)
- Severity: `critical`
- Category: `Diff`
- Description: `inputs.name` / `inputs.request_id` are interpolated directly into shell `run` blocks using `${{ ... }}`. Expression substitution happens before shell execution; payloads containing command substitution syntax can execute unintended commands. This job has `contents: write`, so impact includes repository modification.
- Suggested fix: Stop inlining `${{ inputs.* }}` inside shell commands. Inject via `env:` and reference only quoted `"$VAR"`; add strict allowlist validation for `NAME` and `REQUEST_ID` before use.

### High

2. **Missing AGENTS.md hierarchy and CLAUDE compatibility symlink model**
- File/Dir: `/Users/loopq/dev/git/loopq/trend.github.io` (root and major subdirectories)
- Severity: `high`
- Category: `Holistic`
- Description: No `AGENTS.md` files were found across major directories, and there is no per-directory `CLAUDE.md` symlink pattern tied to agent instructions. Current setup relies on standalone `CLAUDE.md` files only.
- Suggested fix: Add `AGENTS.md` at root and key domains (`scripts/`, `scripts/quant/`, `scripts/backtest/`, `docs/quant/`, `.github/`), and create matching `CLAUDE.md` symlinks for single-source operational guidance.

3. **Input validation in backtest workflow is incomplete and inconsistent**
- File: `.github/workflows/backtest.yml` (lines 20-23, 52-74)
- Severity: `high`
- Category: `Diff`
- Description: Current validation for `name` is blacklist-based and misses shell-relevant metacharacter cases; `request_id` lacks strong validation but is later used in command/log contexts.
- Suggested fix: Replace blacklist with allowlist regex for `name`; enforce strict `request_id` pattern (e.g., UUID or `manual`) and hard-fail invalid input early.

4. **Main production pipeline lacks CI-level automated test gate equivalent to quant module**
- File/Dir: `scripts/` and `.github/workflows/quant-test.yml`
- Severity: `high`
- Category: `Holistic`
- Description: Automated coverage is strong for `scripts/quant/`, but core pipeline (`scripts/main.py`, `data_fetcher.py`, `calculator.py`, `generator.py`) still depends heavily on manual verification, increasing regression risk.
- Suggested fix: Add pytest suites for trading-day gate, stale-data gate, fallback logic, template rendering, and archive generation; add a dedicated required CI workflow for these tests.

5. **Client-side PAT + front-end gate creates weak server/client security boundary**
- File: `docs/quant/lib/writer.js`, `docs/quant/settings.html`, `docs/quant/lib/auth.js`
- Severity: `high`
- Category: `Holistic`
- Description: Browser stores PAT in `localStorage` and writes directly to GitHub API; access control is front-end MD5 gate. Compromised browser/session can directly abuse write capability.
- Suggested fix: Move write operations to controlled server endpoint with short-lived credentials, audit logging, and least privilege. If client flow remains, document threat model and enforce stricter token policy/rotation and monitoring.

### Medium

6. **`git pull --rebase` errors are swallowed in push retry loop**
- File: `.github/workflows/backtest.yml` (line 114)
- Severity: `medium`
- Category: `Diff`
- Description: `git pull --rebase origin main || true` hides rebase failures and may continue to push in broken states, obscuring root cause.
- Suggested fix: Remove `|| true`; fail or retry with explicit rebase-state cleanup and deterministic error handling.
